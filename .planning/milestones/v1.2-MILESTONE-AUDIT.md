### Unsatisfied / Partial Requirements

#### GEAR-01 — Partial (Critical Gap)

**Description:** `IsFreeCarry` throughout model/DTO/service/VM/UI layers; auto-detect; slot exclusion; sub-collections.

**Core layer:** Fully satisfied. `SdCharacterSheet.Core/Models/GearItem.cs` (line 9), `MagicItem.cs` (line 8), `DTOs/CharacterSaveData.cs` (lines 65, 73), `Services/CharacterFileService.cs` (lines 81, 90, 139, 148) — all updated. 3 GEAR-01 unit tests pass.

**MAUI app layer:** **NOT updated.** The following files were last modified in the initial scaffold commit only (`5d04d85 feat(01-01)`) and are missing `IsFreeCarry`:

| File | Expected | Actual |
|------|----------|--------|
| `SdCharacterSheet/Models/GearItem.cs` | `bool IsFreeCarry { get; set; }` | ABSENT |
| `SdCharacterSheet/Models/MagicItem.cs` | `bool IsFreeCarry { get; set; }` | ABSENT |
| `SdCharacterSheet/DTOs/CharacterSaveData.cs` | `IsFreeCarry` on `GearItemData` and `MagicItemData` | ABSENT |
| `SdCharacterSheet/Services/CharacterFileService.cs` | `IsFreeCarry = g.IsFreeCarry` in `MapToDto` (lines 103-110) and `MapFromDto` (lines 156-163) | ABSENT |

**Runtime consequence:** `MauiCharacterFileService` inherits from the MAUI-local `CharacterFileService`. When a character is saved and reloaded through the running app, `IsFreeCarry = true` is silently dropped from the JSON for all items. On load, only items whose names match `KnownFreeCarryNames` (Backpack, Bag of Coins, Thieves Tools) recover their free-carry status via `GearItemViewModel`'s auto-detect fallback. Any item the user manually checked as Free Carry will revert to regular gear after every save/load cycle.

**Potential compile error:** `GearItemViewModel` accesses `g.IsFreeCarry` and `m.IsFreeCarry` where `GearItem`/`MagicItem` are imported via `using SdCharacterSheet.Models`. Both the MAUI-local assembly and Core assembly define `SdCharacterSheet.Models.GearItem`. The local definition wins (CS0436 warning), but the local `GearItem` lacks `IsFreeCarry` — this should produce CS0117 compile errors. The plan executor's MAUI builds failed at the Xcode toolchain stage (actool/xcrun), which may have masked C# compilation errors. **MAUI app build correctness is unverified.**

**Fix required:**
```
SdCharacterSheet/Models/GearItem.cs   → add: public bool IsFreeCarry { get; set; }
SdCharacterSheet/Models/MagicItem.cs  → add: public bool IsFreeCarry { get; set; }
SdCharacterSheet/DTOs/CharacterSaveData.cs → add IsFreeCarry to GearItemData and MagicItemData
SdCharacterSheet/Services/CharacterFileService.cs → add IsFreeCarry = g.IsFreeCarry in MapToDto and MapFromDto
```

## Audit Conclusion

The milestone is **not complete**. GEAR-01 has a critical gap in the MAUI application layer: the MAUI-local models, DTOs, and CharacterFileService are not aware of `IsFreeCarry`. This means the feature works in unit tests and in-session (before save), but will silently lose manually-flagged free-carry items on every save/load cycle in the running app. Additionally, the MAUI app may fail to compile due to the local model lacking the property accessed by `GearItemViewModel`.

GEAR-02 and STAT-01 are fully satisfied. STAT-01 (stat drill-down + Shadowdarklings import) has no gaps. GEAR-02 export parity is correctly wired but depends on GEAR-01 being fixed at the save/load layer for the slot counts to remain consistent after a reload.
